Writing Systems & Unicode

Human languages are spoken, signed, and written. But writing is not a transparent window onto speech — it is a technology, invented independently at least four times in human history, with radically different design principles across cultures. Some systems record individual sounds. Others record entire words. Some read left to right; others right to left; still others in alternating directions like an ox plowing a field.

For anyone working in NLP, understanding writing systems is not academic trivia — it is the foundation of everything. Before a neural network sees a single token, text must be encoded in bytes. And the story of how the world agreed (mostly) on a single encoding standard is one of the great engineering sagas of the 20th century.

What This Tutorial Covers

We will travel from the invention of alphabets to the inner workings of UTF-8, through the encoding wars, Unicode normalization, and the surprising information-theoretic connections between writing systems. By the end you will understand why len("cafe\u0301") returns 5 in Python, and why that matters.

1. Types of Writing Systems

Linguists classify writing systems by what unit of language each symbol represents. The major types form a spectrum from sound-based to meaning-based:

Alphabets

In an alphabet, each symbol (letter) represents roughly one phoneme — a minimal unit of sound. The Latin alphabet used for English has 26 letters; Cyrillic (Russian) has 33; Greek has 24. In a "perfect" alphabet, there would be a one-to-one mapping between letters and sounds. Finnish and Korean come close; English is famously far from it (consider though, through, thought, tough, thorough).

Abjads

An abjad writes only consonants. Vowels are either omitted entirely or optionally indicated with small diacritical marks. Arabic and Hebrew are the primary examples. In Arabic, the word for "book" is written as ktb (كتب), and readers infer the vowels from context. This works because Semitic languages have a root-and-pattern morphology where consonant skeletons carry the core meaning.

Abugidas

In an abugida (also called an alphasyllabary), each symbol represents a consonant with an inherent vowel, typically /a/. Other vowels are indicated by modifying the base symbol with diacritics or appended marks. Devanagari (used for Hindi, Sanskrit), Thai, Ethiopic (Ge'ez), and most South and Southeast Asian scripts are abugidas. For example, in Devanagari: क = /ka/, कि = /ki/, कु = /ku/.

Syllabaries

A syllabary has one symbol per syllable. Japanese hiragana and katakana each contain about 46 basic symbols, one per mora (a unit similar to a syllable). Cherokee, developed by Sequoyah in the 1820s, is another example with 85 characters. Syllabaries work well for languages with simple syllable structures (mostly CV — consonant + vowel).

Logographies

In a logographic system, each symbol represents a morpheme or word. Chinese hanzi is the most prominent example, with tens of thousands of characters (a literate adult knows 6,000–8,000). Japanese kanji are borrowed Chinese characters used alongside the syllabaries. Despite the name, most "logographic" characters actually contain phonetic clues — about 80% of Chinese characters have a semantic radical and a phonetic component.

Mixed Systems

Japanese is the world champion of mixed writing. A single sentence may contain:

  • Kanji (logographic) for content words
  • Hiragana (syllabary) for grammatical particles and native words
  • Katakana (syllabary) for foreign loanwords and emphasis
  • Romaji (Latin alphabet) for acronyms and styling

Writing Direction

Not all scripts flow left-to-right. Arabic and Hebrew are written right-to-left (RTL). Traditional Chinese and Japanese run top-to-bottom, right-to-left in columns. Ancient Greek was sometimes written in boustrophedon — alternating direction on each line, like an ox plowing a field (the word literally means "ox-turning"). Even the individual letters were mirrored on alternate lines.

2. From ASCII to Unicode: A Brief History

The story of text encoding is a story of growing ambition — from 128 characters to nearly 155,000, from one language to every script ever written.

ASCII (1963): The 7-Bit Beginning

The American Standard Code for Information Interchange used 7 bits to encode 128 characters: 26 uppercase letters, 26 lowercase, 10 digits, 33 control characters, and a handful of punctuation marks. It was designed by and for English speakers. The letter A is 65 (binary 1000001), a is 97 (1100001), and the space character is 32.

The 8th Bit Problem

Computers use 8-bit bytes, but ASCII only needs 7 bits. That spare bit became a battleground. Every country, every vendor, every operating system used the extra 128 slots (128–255) differently. The result: chaos.

The Code Page Era: Babel in Binary

The explosion of incompatible 8-bit encodings reads like a roll call of regional computing history:

  • ISO 8859-1 (Latin-1): Western European languages. The de facto default of early web.
  • Windows-1252: Microsoft's superset of Latin-1 with "smart quotes" and em-dashes.
  • KOI8-R: Russian Cyrillic, designed so that if the 8th bit was stripped, the text would be transliterated into (garbled but readable) Latin letters.
  • Shift_JIS: Japanese, using a variable-length scheme to cram kanji into bytes.
  • Big5: Traditional Chinese (Taiwan, Hong Kong).
  • GB2312 / GBK: Simplified Chinese (mainland China).

The Mojibake Problem

When text encoded in one system was decoded with another, the result was mojibake (文字化け) — a Japanese word meaning "character transformation." The French word résumé encoded in Latin-1 and decoded as Windows-1252 might display as rÃ©sumÃ©. Russian text decoded as Latin-1 produced streams of accented Latin characters that looked like a European language having a seizure.

Unicode (1991): One Encoding to Rule Them All

The Unicode Consortium set out to assign a unique number — a code point — to every character in every script, living or dead. The first version (1991) covered 7,161 characters. As of Unicode 16.0, the standard defines 154,998 characters from 168 scripts, including Egyptian hieroglyphs, Braille, musical notation, mathematical symbols, and of course emoji.

Unicode organizes its code space into 17 planes, each containing 65,536 code points:

  • Plane 0 (U+0000–U+FFFF): The Basic Multilingual Plane (BMP) — contains the vast majority of common characters from all modern scripts.
  • Plane 1 (U+10000–U+1FFFF): Supplementary Multilingual Plane — historic scripts, emoji, musical and mathematical symbols.
  • Plane 2 (U+20000–U+2FFFF): CJK Unified Ideographs Extension B — rare Chinese/Japanese/Korean characters.
  • Planes 3–13: Mostly unassigned (room for growth).
  • Plane 14: Tags and variation selectors.
  • Planes 15–16: Private Use Areas.

3. UTF-8: The Elegant Variable-Length Encoding

Unicode assigns numbers to characters, but it says nothing about how those numbers are stored as bytes. That is the job of encoding forms. Three exist — UTF-8, UTF-16, and UTF-32 — but UTF-8 has won decisively. It is used by over 98% of all web pages.

UTF-8 was designed in 1992 by Ken Thompson and Rob Pike (of Unix and Go fame) on a placemat in a New Jersey diner. Its genius lies in being variable-length and backward-compatible with ASCII:

UTF-8 Encoding Rules
Code Point Range Bytes Byte Pattern Covers
U+0000–U+007F 1 0xxxxxxx ASCII (English, digits, basic punctuation)
U+0080–U+07FF 2 110xxxxx 10xxxxxx Latin extended, Greek, Cyrillic, Arabic, Hebrew
U+0800–U+FFFF 3 1110xxxx 10xxxxxx 10xxxxxx CJK ideographs, Devanagari, Thai, most BMP
U+10000–U+10FFFF 4 11110xxx 10xxxxxx 10xxxxxx 10xxxxxx Emoji, historic scripts, musical notation, rare CJK
Encoding Example: The Euro Sign

The euro sign € is U+20AC. In binary: 0010 0000 1010 1100. This falls in the 3-byte range (U+0800–U+FFFF), so we use the pattern 1110xxxx 10xxxxxx 10xxxxxx:

  1. Split the bits into groups: 0010 | 000010 | 101100
  2. Fill the template: 11100010 10000010 10101100
  3. In hex: E2 82 AC

So the single character € becomes three bytes in UTF-8. You can verify: "€".encode("utf-8") returns b'\xe2\x82\xac' in Python.

Why UTF-8 Won

UTF-8's dominance is no accident. It has several remarkable properties:

  • Backward compatible with ASCII: any valid ASCII file is also valid UTF-8, byte for byte. This made migration painless.
  • Self-synchronizing: you can jump to any byte and find the start of the next character by looking for a byte that does not start with 10. No need to scan from the beginning of the file.
  • No byte-order issues: unlike UTF-16 and UTF-32, there is no endianness ambiguity. No BOM needed.
  • Compact for ASCII text: English text takes exactly one byte per character, same as ASCII.
  • No embedded null bytes: the only character that produces a 0x00 byte is the actual null character. This means UTF-8 strings work with C string functions.

4. Unicode Normalization: The Hidden Complexity

Unicode contains a subtle trap that has caused countless bugs: the same visual character can be represented by different sequences of code points.

Composed vs. Decomposed Forms

Consider the character é (e with acute accent). Unicode offers two representations:

  • Composed (precomposed): U+00E9 — a single code point for é
  • Decomposed: U+0065 U+0301 — the letter e followed by a "combining acute accent"

They look identical when rendered. But they are different byte sequences: 

# In Python:
composed = "\u00e9"        # é (1 code point)
decomposed = "e\u0301"     # e + combining accent (2 code points)

composed == decomposed     # False!
len(composed)              # 1
len(decomposed)            # 2
The Four Normalization Forms
Form Name What It Does
NFC Canonical Decomposition + Composition Decomposes, then recomposes into precomposed forms where possible. Most common for the web.
NFD Canonical Decomposition Fully decomposes into base + combining characters. macOS uses this for filenames.
NFKC Compatibility Decomposition + Composition Like NFC, but also replaces compatibility characters (e.g., ﬁ → fi). Lossy but useful for search.
NFKD Compatibility Decomposition Fully decomposes including compatibility characters. Maximum decomposition.

Why This Matters for NLP

If you do not normalize text before processing, two "identical" strings may not match in search, may produce different hashes, may tokenize differently, and may confuse downstream models. The standard practice in NLP pipelines is to normalize to NFC (or NFKC for tasks that need compatibility equivalence like search).

Homoglyphs: When Different Characters Look the Same

Normalization handles cases where the same character has multiple representations. But a related problem is when different characters look identical:

  • Latin a (U+0061) vs. Cyrillic а (U+0430) — visually indistinguishable in most fonts
  • Latin o (U+006F) vs. Greek ο (U+03BF, omicron)
  • Latin p (U+0070) vs. Cyrillic р (U+0440)

This is a serious security concern. An attacker could register a domain like аpple.com (with Cyrillic a) that looks identical to apple.com in a browser's address bar. This class of attack is called a homoglyph attack or IDN homograph attack.

5. Challenges for NLP Across Writing Systems

Most NLP tools were built with English (or at least alphabetic scripts) in mind. Extending them to the full diversity of the world's writing systems reveals deep challenges:

Word Segmentation

English conveniently puts spaces between words. Chinese, Japanese, and Thai do not. The string 今天天气很好 ("The weather is very nice today") is written without spaces. Segmenting it into words (今天 / 天气 / 很 / 好) requires a model — often a dictionary-based or statistical segmenter. Getting this wrong cascades errors through every downstream task.

Bidirectional Text

Real documents mix scripts with different directions. An Arabic sentence may contain an English brand name, a number, or a URL. The Unicode Bidirectional Algorithm (UAX #9) defines rules for rendering such mixed text, but it introduces surprising edge cases. For example, parentheses and punctuation marks must be "mirrored" depending on context. This is one of the trickiest parts of text rendering.

Case

Not all scripts have the concept of uppercase and lowercase. Chinese, Japanese, Korean, Arabic, Hebrew, Thai, and most other scripts have no case distinction. NLP tools that assume case-folding (lowercasing) as a preprocessing step may need script-aware logic. And some scripts have unexpected case behavior: German ß uppercases to SS (one character becomes two).

Script Detection

Given a string of text, which script is it in? For single-script text this is straightforward (check the Unicode block). But for mixed-script text, or for scripts that share characters, detection requires statistical methods. Unicode provides Script properties for every character, but the practical challenge is handling multilingual documents gracefully.

Transliteration

Converting between scripts — for example, Cyrillic to Latin (romanization) or Latin to Devanagari — is a lossy, context-dependent process. There is no single correct romanization of Russian: Щ can be transliterated as shch, sch, or šč depending on the system used. Machine transliteration is an active research area, especially for named entities.

6. The Information-Theoretic View

Different writing systems represent the same information using different numbers of symbols with different frequencies. Information theory gives us a precise way to compare them.

Entropy Per Character

The entropy of a character in a writing system depends on the size of the alphabet and the frequency distribution of characters:

$$H = -\sum_{i=1}^{N} p_i \log_2 p_i$$

For a uniform distribution over $N$ symbols, the maximum entropy is $\log_2 N$:

Entropy Per Character Across Writing Systems
Writing System Alphabet Size $N$ Max Entropy $\log_2 N$ Empirical $H$ (bits/char)
Binary 2 1.00 1.00
English (26 letters) 27 (+ space) 4.76 ~4.08
Russian (Cyrillic) 34 (+ space) 5.13 ~4.35
Japanese (hiragana) ~46 5.52 ~4.7
Chinese (common hanzi) ~6,000 12.55 ~9.7

The Constant Information Rate Hypothesis

Here is a remarkable fact: despite the enormous variation in entropy per character, the information transmitted per second of reading (or speaking) is roughly constant across languages, at about 39 bits per second.

Languages with low-entropy characters (like English) compensate by using more characters per word and more words per sentence. Languages with high-entropy characters (like Chinese) pack more information into each character and use fewer of them. The entropy per character and the characters per second are inversely related:

$$\text{Information rate} = H(\text{char}) \times \text{chars/second} \approx \text{const}$$
Connection to Compression

UTF-8 is, in a sense, a Huffman-like code for Unicode. The most common characters (ASCII, used in URLs, code, English text) get the shortest encoding (1 byte). Less common characters get progressively longer encodings. This is exactly the principle behind Huffman coding: assign shorter codes to more frequent symbols. The key difference is that UTF-8's design is based on code point ranges rather than empirical frequency, but the effect is similar for the global distribution of characters on the web.

7. Practical Python Examples

Let us put theory into practice. Python 3 uses Unicode strings natively, making it an excellent tool for exploring these concepts.

Detecting and Converting Encodings

import chardet

# Detect encoding of raw bytes
raw_bytes = b'\xc3\xa9\x74\x75\x64\x69\x61\x6e\x74'
result = chardet.detect(raw_bytes)
print(result)
# {'encoding': 'utf-8', 'confidence': 0.87, 'language': ''}

# Decode with the detected encoding
text = raw_bytes.decode(result['encoding'])
print(text)  # étudiant

# Convert between encodings
text = "Résumé"
latin1_bytes = text.encode('latin-1')
utf8_bytes = text.encode('utf-8')
print(f"Latin-1: {latin1_bytes}  ({len(latin1_bytes)} bytes)")
# Latin-1: b'R\xe9sum\xe9'  (6 bytes)
print(f"UTF-8:   {utf8_bytes}  ({len(utf8_bytes)} bytes)")
# UTF-8:   b'R\xc3\xa9sum\xc3\xa9'  (8 bytes)

Unicode Normalization

import unicodedata

# The two forms of é
composed = "\u00e9"          # NFC form
decomposed = "e\u0301"       # NFD form

print(f"Composed:   {composed!r}  len={len(composed)}")
# Composed:   'é'  len=1
print(f"Decomposed: {decomposed!r}  len={len(decomposed)}")
# Decomposed: 'e\u0301'  len=2

# They look the same but aren't equal
print(f"Equal? {composed == decomposed}")  # False

# Normalize both to NFC
nfc_1 = unicodedata.normalize('NFC', composed)
nfc_2 = unicodedata.normalize('NFC', decomposed)
print(f"After NFC: {nfc_1 == nfc_2}")  # True

# NFKC goes further: compatibility decomposition
ligature = "\ufb01"  # ﬁ ligature
print(f"NFKC(ﬁ) = {unicodedata.normalize('NFKC', ligature)!r}")
# NFKC(ﬁ) = 'fi'

Characters vs. Bytes

# Different scripts, different byte costs
examples = {
    "English":  "Hello",
    "Russian":  "\u041f\u0440\u0438\u0432\u0435\u0442",
    "Chinese":  "\u4f60\u597d",
    "Arabic":   "\u0645\u0631\u062d\u0628\u0627",
    "Emoji":    "\U0001f600\U0001f44b",
}

print(f"{'Script':<10} {'Text':<10} {'Chars':>5} {'UTF-8 Bytes':>11} {'Bytes/Char':>10}")
print("-" * 50)
for script, text in examples.items():
    chars = len(text)
    byte_len = len(text.encode('utf-8'))
    print(f"{script:<10} {text:<10} {chars:>5} {byte_len:>11} {byte_len/chars:>10.1f}")

# Output:
# Script     Text       Chars UTF-8 Bytes Bytes/Char
# --------------------------------------------------
# English    Hello          5           5        1.0
# Russian    Привет         6          12        2.0
# Chinese    你好             2           6        3.0
# Arabic     مرحبا          5          10        2.0
# Emoji      😀👋             2           8        4.0

The Emoji Trap: Grapheme Clusters

# A family emoji is multiple code points joined by ZWJ (Zero Width Joiner)
family = "\U0001F468\u200D\U0001F469\u200D\U0001F467\u200D\U0001F466"
print(f"Visual: {family}")
print(f"Code points: {len(family)}")         # 7 (4 emoji + 3 ZWJ)
print(f"UTF-8 bytes: {len(family.encode())}")  # 25

# Breakdown:
for i, char in enumerate(family):
    name = unicodedata.name(char, "ZWJ" if char == "\u200d" else "?")
    print(f"  U+{ord(char):04X}  {name}")
# U+1F468  MAN
# U+200D   ZWJ
# U+1F469  WOMAN
# U+200D   ZWJ
# U+1F467  GIRL
# U+200D   ZWJ
# U+1F466  BOY

# What you see as ONE emoji is actually 7 code points and 25 bytes!
# For correct "visual length", you need a grapheme cluster library:
# pip install grapheme
import grapheme
print(f"Grapheme clusters: {grapheme.length(family)}")  # 1
Key Takeaway for NLP Practitioners

In Unicode, there are four different notions of "length": (1) bytes, (2) code units (relevant for UTF-16), (3) code points, and (4) grapheme clusters (what a human perceives as a "character"). These can differ dramatically. A single visible emoji can be 25 bytes, 7 code points, but just 1 grapheme cluster. Always be explicit about which "length" you mean.

Homoglyph Detection

# Are these the same? They look identical but aren't.
latin_a = "a"       # U+0061, Latin Small Letter A
cyrillic_a = "\u0430"  # U+0430, Cyrillic Small Letter A

print(f"Latin a:    U+{ord(latin_a):04X}")
print(f"Cyrillic a: U+{ord(cyrillic_a):04X}")
print(f"Equal? {latin_a == cyrillic_a}")  # False
print(f"Look alike? Yes!")

# Script detection using unicodedata
for char in [latin_a, cyrillic_a]:
    script = unicodedata.name(char)
    print(f"  '{char}' -> {script}")
# 'a' -> LATIN SMALL LETTER A
# 'а' -> CYRILLIC SMALL LETTER A
